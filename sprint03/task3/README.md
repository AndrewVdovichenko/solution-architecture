# Задание 3. Целевая облачная архитектура

## Архитектурное решение

Cloud POS становится тонким клиентом и работает только через стабильные API.
Специфика локального оборудования вынесена в Edge Device Gateway с Hardware
Abstraction Layer (HAL). Для каждой модели терминала или фискального регистратора
разрабатывается отдельный адаптер, но контракт между POS и HAL не меняется.

Постоянная локальная серверная БД удаляется. На edge-устройстве остаются только
зашифрованный минимальный кэш для работы офлайн и transactional outbox для
последующей синхронизации.

## Как решены ключевые ограничения

**Разнородное оборудование.** Web POS вызывает локальный HAL через защищённый API.
HAL выбирает адаптер для конкретной модели ККТ или банковского
терминала. Добавление новой модели не требует пересборки Cloud POS.

**Потеря интернета.** Service Worker/PWA и Edge Gateway используют
снимок минимального каталога и локальную очередь операций. Наличная продажа и
фискальный чек доступны, а функции, требующие онлайн-авторизации, явно
ограничиваются. После восстановления связи очередь синхронизируется идемпотентно.

## Этапы перехода

1. Создать API-слой перед SAP, Promo Engine и Smart Fulfillment.
2. Реализовать Edge HAL и проверить адаптеры на репрезентативном наборе устройств.
3. Провести пилот Cloud POS в 5–10 магазинах, сохраняя старый POS в качестве запасного варианта.
4. Мигрировать магазины волнами, контролируя офлайн-сценарии и сверку операций.
5. После успешной сверки вывести локальные серверы и лицензии из эксплуатации.
