# Матрица Buy vs Build

## Критерии решения

- **Buy** — стандартизированный процесс, не создающий конкурентного
  преимущества.
- **Build** — уникальная логика или интеграционный компонент,
  который должен независимо развиваться и масштабироваться.
- **In-house** — существующая критичная система, миграция которой
  не нужна для достижения целей кейса.
- **Retire** — дублирующая или дорогая возможность, которую следует вывести из
  эксплуатации.

## Матрица решений

| Возможность | Текущая проблема | Дифференциатор | Целевая модель | Обоснование |
| --- | --- | --- | --- | --- |
| POS, чеки, возвраты | Толстый клиент и отдельная БД в магазине | Нет | Buy SaaS Cloud POS | Стандартная функция; покупка сокращает срок внедрения и стоимость поддержки |
| Локальные серверы и БД | Более 500 серверов и лицензий | Нет | Retire | Постоянная локальная БД заменяется облачным решением и минимальным локальным кэшем |
| Драйверы терминалов и ККТ | Пересборка POS под каждую модель | Нет | Build Edge HAL on PaaS/Edge | Единый контракт HAL изолирует POS от конкретных драйверов |
| Платёжная интеграция | Жёсткая связь POS с терминалом | Нет | Buy SaaS платёжный шлюз + HAL | Провайдер стандартизирует эквайринг, HAL обеспечивает доступ к локальному устройству |
| Лояльность | Недоступна офлайн, ведётся локально вручную | Нет | Buy SaaS Система лояльности | Стандартный централизованный процесс с API для POS |
| HR, заработная плата и премии | Локальный расчёт в магазинах | Нет | Buy SaaS 1С | Исключает дублирование и локальные расчёты |
| Мониторинг устройств | Реактивная поддержка и выезды | Нет | Buy SaaS Device Management | Централизованные обновления, health-check и удалённая диагностика |
| SKU Matching | До 80% ручной проверки, срок до двух недель | Частично | Build on PaaS + AI API | Специфичный для M&A процесс; нужен контролируемый человеком процесс |
| Master Data / SKU | Разрозненные справочники | Нет | In-house SAP ERP | SAP остаётся источником истины, API-слой убирает файловый обмен |
| Promo Engine | Уже централизован | Частично | In-house | Сохранить правила и предоставить стабильный API для Cloud POS |
| Smart Fulfillment | Ключевое преимущество сети | Да | In-house | Уникальная логика должна оставаться под контролем VitaStore |
| Click & Collect | Часть омниканального преимущества | Да | In-house | Тесно связано со Smart Fulfillment |
| SAP ERP | Критичная центральная система | Нет | In-house | Переписывание не требуется; интеграция через API снижает риск |
| Приёмка и запасы | Разные локальные процессы | Нет | Buy | Стандартизация ускоряет подключение купленных магазинов |
| M&A онбординг | Email, Excel и ручная координация | Частично | Build | Оркестрирует проверки, SLA, аудит и интеграцию всех систем |

## Последовательность

1. Внедрить портал онбординга, SKU Matching и интеграционный API-слой.
2. Выбрать Cloud POS, Payment Gateway, Device Management и Edge HAL.
3. Провести пилот на 5–10 магазинах с разным оборудованием.
4. Мигрировать магазины волнами и выводить локальные серверы из эксплуатации.
