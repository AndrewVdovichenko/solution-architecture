# AS-IS анализ текущей архитектуры AdScale

## Текущее состояние

AdScale — монолит, построенный на C++, Go, Python и Node.js. Все
сервисы используют один и тот же инстанс базы данных PostgreSQL. Кеш,
очереди, реплики на чтение, контейнеризация и автоматизированный деплой отсутствуют.

Платформа уже работает на пределе:

- пиковая нагрузка — 15 000 RPS;
- задержка — 110-120 ms при требовании не более 100 ms;
- периодические деградации в часы пик;
- отсутствует запас перед подключением новой DSP, которая добавит около 20%
  трафика и выведет нагрузку примерно на 18 000 RPS.

## Критический путь запроса ставки

Критичный путь выглядит так:

1. DSP-партнер отправляет bid запрос в Ad Server.
2. Ad Server определяет пользователя и подбирает кандидатов.
3. Auction Engine применяет бизнес-правила и выбирает победителя.
4. Delivery Service формирует ответ.
5. Statistics Service регистрирует показы и клики.

Основная проблема: синхронные обращения к базе данных в критичном пути.
При росте нагрузки это увеличивает задержки и создает риск таймаутов на стороне DSP.

## Узкие места

| Область | Проблема | Влияние |
| --- | --- | --- |
| PostgreSQL | Одна реплика для всего | Конкуренция запросов, блокировки, рост задержек |
| Auction Engine | Синхронные вызовы в критическом пути | Невозможность стабильно удерживать ответ до 80-100 ms |
| Statistics Service | Прямая запись кликов и показов в PostgreSQL | Становится узким горлышком для записи |
| Analytics Service | Тяжелые SQL-запросы к БД | Блокировки, деградация кабинета и ставок |
| Ad Server | Нет кеша кампаний, таргетинга и ставок | Лишние запросы в базу данных на каждый bid request |
| Деплой | Ручной релиз без контейнеризации | Долгие изменения, высокий риск простоя |
| Наблюдаемость | Не описаны SLO, трассировки, бизнес-метрики | Сложно управлять задержками и штрафами DSP |

## Точки отказа

- PostgreSQL — единая точка отказа для ставок, финансов, аналитики и кабинета.
- Монолит — сбой некритичного модуля может повлиять на критичный аукцион.
- Ручной деплой — риск ошибки и длительного восстановления.
- Отсутствие очередей — всплески событий напрямую давят на БД.
- Отсутствие API Gateway для DSP — нет централизованного узла для аутентификации, маршрутизации, ограничения запросов.

## Архитектурные проблемы

1. **Смешение критичных и некритичных потоков.** Аукцион, аналитика, финансы,
   кабинет и события используют общие ресурсы.
2. **Синхронность в критичном пути.** Запрос ставки зависит от доступности и скорости
   PostgreSQL.
3. **Невозможность независимого масштабирования.** Нельзя отдельно увеличить
   Auction Engine, Statistics Service или Analytics Service.
4. **Отсутствие backpressure.** Поток кликов и показов не буферизуется.
5. **Слабая отказоустойчивость.** Нет High Availability для БД, брокера событий и
   автоматизированного восстановления.
6. **Недостаточная готовность к интеграции DSP.** Новые SLA строже текущего
   фактического P95.

## Вывод

Текущая архитектура не выдержит подключение новой DSP без риска деградации.
Главные изменения на ближайшие три месяца должны уменьшить задержки в критичном пути,
изолировать события от OLTP-БД, добавить кеширование и ограничить влияние
аналитики на нагрузку на базу данных.
