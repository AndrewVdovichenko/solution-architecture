# Архитектурные драйверы

## Функциональные требования

- Принимать bid запросы от новой DSP-площадки по OpenRTB over HTTPS.
- Формировать bid ответы в пределах SLA партнера.
- Поддерживать не менее 18 000 RPS через три месяца без деградации текущих
  клиентов.
- Сохранять клики, показы, bid запросы и bid ответы для статистики,
  биллинга и аналитики.
- Предоставлять рекламодателям кабинет для управления кампаниями, бюджетами,
  ставками и отчетностью.
- Обеспечивать корректные учёт и списание средств, финансовую сверку.
- Поддерживать постепенную миграцию без остановки бизнеса.

## Атрибуты качества

| Приоритет | Атрибут | Целевое состояние |
| --- | --- | --- |
| 1 | Latency | P95 не выше 100 ms для аукциона через 3 месяца; стремиться к запасу 80 ms для DSP |
| 1 | Scalability | 18 000 RPS через 3 месяца, 50 000 RPS через год |
| 1 | Reliability | Сбой аналитики или статистики не должен ломать выдачу ставок |
| 1 | Availability | SLA 99,9% на горизонте года |
| 2 | Observability | Метрики latency, error rate, timeout rate, auction win/loss, Kafka lag, DB saturation |
| 2 | Maintainability | Независимое развитие критичных сервисов без полной переписи монолита |
| 2 | Data consistency | Строгая консистентность для финансов, событийная консистентность для статистики и аналитики |
| 3 | Cost efficiency | Использовать поэтапную миграцию и стандартные компоненты без большой платформенной команды |

## Бизнес-ограничения

- **Срок** промежуточного решения — три месяца.
- **Бюджет** ограничен, полной переписи быть не должно.
- Необходимо сохранить **непрерывность** бизнеса и текущих клиентов.
- **Команда**: 35 человек, без выделенных SRE, технического писателя и
  бизнес-аналитика.
- Новая DSP вводит **штрафы** за задержку и простой.

## Технические ограничения

- Исходная система — монолит на C++, Go, Python и Node.js.
- PostgreSQL используется как единая БД.
- Контейнеризация и автоматизированный деплой отсутствуют.
- Нет кеша, брокера сообщений и реплики для чтения.
- Необходима постепенная миграция, сохраняющая совместимость с существующим кабинетом
  и финансовым контуром.

## Архитектурные принципы

- Критичный путь должен быть коротким, предсказуемым. Иметь минимальную зависимость
  от синхронных внешних вызовов.
- Клики, показы и аналитика должны обрабатываться асинхронно.
- Финансовые операции должны оставаться транзакционными и идемпотентными.
- Каждый новый сервис должен иметь собственную ответственность, собственные
  метрики. Иметь возможность независимого масштабирования.
- Монолит не удаляется сразу, а постепенно теряет ответственность через паттерн
  Strangler Fig.
