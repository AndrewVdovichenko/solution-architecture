# TO-BE архитектура AdScale

## Целевое видение

Через год AdScale должна перейти к микросервисной архитектуре с выделенным RTB
контуром, событийной обработкой данных, горизонтальным масштабированием и
разделением хранилищ по сервисам.

Целевые показатели:

- до 50 000 RPS;
- P95 задержки аукциона не выше 100 ms;
- поддержка нескольких DSP-партнеров;
- независимое масштабирование Bidding, Statistics, Analytics и Campaign
  сервисов;
- SLA 99,9%;
- отказоустойчивое хранение данных и потоков событий.

## Архитектурный стиль

Оптимальный стиль — постепенная декомпозиция с помощью паттерна Strangler Fig.

Оставаться на монолите нельзя потому, что

- текущий P95 уже выше требования;
- общий PostgreSQL не масштабируется под RTB, аналитику и события одновременно;
- сбой некритичных модулей влияет на критичный путь ставок;
- невозможно отдельно масштабировать Auction Engine и запись событий.

При этом не получится перейти сразу на микросервисы, так как

- бизнес требует непрерывности работы;
- срок три месяца не позволяет безопасно переписать и протестировать всю платформу;
- высокая вероятность регрессий в финансах и кабинете;
- в команде нет выделенного SRE.

Предлагается последовать по практичному пути из нескольких этапов.

1. Вынести Bidding Service и поставить его за API Gateway.
2. Добавить Redis для горячих данных и Kafka для событий.
3. Изолировать аналитику от базы данных PostgreSQL.
4. Постепенно выделить сервисы: Campaign, Statistics, Analytics и Finance.
5. Перейти к подходу Database per Service и независимому масштабированию.

## Границы сервисов

| Сервис | Ответственность | Хранилище |
| --- | --- | --- |
| API Gateway | DSP auth, routing, rate limiting, circuit breaker, latency metrics | Конфигурация gateway |
| Bidding Service | Принятие bid запросов, выбор ставки, формирование bid ответов | PostgreSQL, Redis |
| Campaign Service | Кампании, ставки, таргетинг, статусы | PostgreSQL |
| Delivery Service | Подготовка HTML-разметки | Object storage, cache |
| Statistics Service | Прием и агрегация кликов, показов, bid events | Kafka, ClickHouse |
| Analytics Service | Отчеты и тяжелые запросы | ClickHouse |
| Finance Service | Балансы, списания, счета, сверки | PostgreSQL |
| Advertiser Dashboard | UI и BFF для рекламодателей | Через сервисные API |

## Основные взаимодействия

- DSP отправляет bid запрос в API Gateway по OpenRTB over HTTPS.
- API Gateway проверяет партнера, применяет ограничения на RPS и направляет запрос в
  Bidding Service.
- Bidding Service использует Redis для кампаний, ставок, бюджетных лимитов и
  таргетинга.
- Campaign Service публикует изменения кампаний в Kafka, Bidding Service
  обновляет кеш.
- Advertiser Dashboard работает через сервисные API и читает отчеты из
  аналитических read models.

## Промежуточная архитектура на 3 месяца

Минимальный объем изменений для подключения DSP:

- Внедрить API Gateway перед RTB-контуром.
- Выделить Bidding Service.
- Внедрить Redis.
- Выделить отдельный аналитический контур для тяжелых отчетов (например, реплика БД для чтения).
- Создать дашборд SLO: latency, errors, timeouts, DB saturation.
