# ADR-001. Стратегия эволюции архитектуры

## Контекст

AdScale работает как монолит. Система уже не выполняет требование по задержкам: фактическое значение 110-120 ms при целевом уровне не выше 100 ms.
Новая DSP добавит около 20% нагрузки, а за год ожидается рост до 50 000 RPS.

Полная перепись за три месяца невозможна из-за ограниченного срока, бюджета и
риска остановки бизнеса.

## Решение

Использовать паттерн Strangler Fig: постепенно выносить из монолита наиболее
нагруженные и критичные контуры, сохраняя работоспособность существующей
платформы.

Порядок декомпозиции:

1. Bidding Service и API Gateway для RTB-трафика.
2. Kafka для событий кликов, показов и ставок.
3. Statistics и Analytics сервисы с отдельным OLAP-хранилищем.
4. Campaign Service и Finance Service с отдельными БД.

## Последствия

Плюсы:

- снижение риска полной переписи;
- быстрый эффект для снижения задержек;
- возможность независимого масштабирования критичных сервисов;
- сохранение непрерывности бизнеса.

Минусы:

- потребуется синхронизация данных между монолитом и новыми сервисами;
- временно возрастает сложность интеграций.
