# Взаимодействие и протоколы

## Общая схема

RTB-контур разделяется на синхронный критичный путь и асинхронную обработку
событий.

Синхронный путь:

1. DSP -> API Gateway: OpenRTB over HTTPS.
2. API Gateway -> Bidding Service: gRPC.
3. Bidding Service -> Redis: чтение горячих данных.
4. Bidding Service -> API Gateway -> DSP: bid/no-bid response.

Асинхронный путь:

1. Bidding Service публикует bid events в Kafka.
2. Delivery Service публикует показы/клики в Kafka.
3. Statistics, Analytics и Finance сервисы читают свои события.

## Выбор протоколов

| Связь | Протокол | Обоснование |
| --- | --- | --- |
| DSP -> API Gateway | HTTPS / OpenRTB | Отраслевой стандарт RTB-интеграций |
| API Gateway -> Bidding | gRPC | Низкая задержка, строгий контракт, бинарная сериализация |
| Dashboard -> сервисы | REST | Удобно для UI/BFF, не является критичным путём |
| Campaign -> Bidding cache | Kafka events | Изменения кампаний должны обновлять кеш асинхронно |
| Clicks/impressions/bids | Kafka | Потоковые события, backpressure, независимые потребители |
| Finance operations | REST/gRPC + events | Синхронно для команд, асинхронно для событий биллинга |

## Почему gRPC для Bidding

- Контракт строго типизирован и удобен для генерации клиентов.
- Низкие накладные расходы по сравнению с JSON REST.
- Хорошо подходит для внутренних сервисных вызовов с жестким лимитом на задержки.

## Почему REST для Campaign и Dashboard

- Операции управления кампаниями не находятся на критичном пути.
- REST проще для BFF и фронтенд-приложений.
- Удобно версионировать административные API.

## Почему Kafka для событий

- Клики, показы и ставки являются высокочастотным потоком событий.
- Kafka сглаживает пики и защищает OLTP-хранилища от интенсивной записи.
- Разные потребители могут обрабатывать один поток независимо.
- Retention позволяет переобрабатывать данные для аналитики и сверки.

## Нужен ли API Gateway

Да. Для DSP-интеграции API Gateway нужен как отдельный защитный и
маршрутизирующий слой:

- аутентификация;
- маршрутизация на правильную версию Bidding Service;
- квоты и ограничения RPS;
- централизованное управление таймаутами;
