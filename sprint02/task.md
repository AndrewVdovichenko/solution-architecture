Проведите анализ, чтобы выявить узкие места и точки отказа, определить архитектурные проблемы, препятствующие масштабированию, и оценить влияние текущей архитектуры на время отклика.

## Задание 2. Проектирование RTB и интеграции с DSP

**Что нужно сделать**: спроектировать архитектуру критичного компонента — RTB-сервиса для интеграции с новой DSP-площадкой с соблюдением требований по времени отклика и надёжности.

### 1. Проектирование сервиса ставок
Выделите сервис ставок из монолита, опишите его границы и API, определите зависимости от других сервисов и спроектируйте его модель данных.

### 2. Проектирование взаимодействия

Определите протоколы и паттерны взаимодействия. Для внутреннего — выбор между REST, gRPC и асинхронным подходом. Для внешнего — протокол интеграции с DSP. Для потоковой обработки событий — Kafka для событий кликов и показов.

Обоснуйте, почему gRPC или REST для Bidding и Campaign, а Kafka для событий? API Gateway нужен? Если да, то какой?

### 3. API Gateway для DSP-интеграции

Спроектируйте API-слой для новой DSP-площадки: маршрутизацию запросов на ставки, ограничение частоты запросов, аутентификацию, паттерн автоматического размыкателя и мониторинг времени отклика.

### 4. Паттерны надёжности

Опишите, как обеспечить надёжность: паттерн Circuit Breaker для вызовов зависимых сервисов, повторные попытки с экспоненциальной задержкой, идемпотентность финансовых операций и резервные стратегии (кэшированные или дефолтные ставки).

В результате у вас должно получиться:

- README.md — краткое описание;
- bidding-service.md — спецификация сервиса ставок;
- interaction.md — схема взаимодействия с обоснованием выбора протоколов;
- api-gateway.md — дизайн API Gateway для DSP;
- diagrams/ — диаграммы взаимодействия (sequence, component).
