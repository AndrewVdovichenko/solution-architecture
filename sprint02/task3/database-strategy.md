# Database per Service

## Выбор базы данных

AdScale переходит от единой базы данных PostgreSQL к Database per Service.
Такой подход позволит уменьшить связанность, масштабировать домены независимо и защитить
критичный RTB-путь от аналитической и событийной нагрузки.

| Сервис | Тип базы данных | Обоснование |
| --- | --- | --- |
| Сервис ставок | PostgreSQL для конфигурации, Redis для горячих данных | Bid decision должен читать данные за миллисекунды, для этого подойдёт Redis; PostgreSQL не используется в критичном пути |
| Сервис кампаний | PostgreSQL | Кампании, таргетинг и ставки требуют транзакционной целостности и удобной реляционной модели |
| Сервис статистики | Kafka + ClickHouse | Высокий поток кликов/показов, быстрые агрегации |
| Сервис аналитики | ClickHouse | Тяжелые отчеты должны быть изолированы от OLTP и RTB. ClickHouse хорошо подходит для быстрых выборок данных за разные периоды |
| Сервис финансов | PostgreSQL | Балансы, списания и счета требуют ACID-транзакций и строгой консистентности |

## Миграционный подход

1. Оставить текущий PostgreSQL как единый источник правды на первом этапе.
2. Вынести аналитический контур для отчетов.
3. Добавить Kafka и начать публиковать события.
4. Выделить Campaign и Finance БД при декомпозиции сервисов.
5. Перевести Bidding на Redis и собственную конфигурационную БД.
