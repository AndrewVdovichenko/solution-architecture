# Отказоустойчивость данных

## RPO и RTO

| Сервис | RPO | RTO | Обоснование |
| --- | --- | --- | --- |
| Bidding | 1-5 минут для config/audit, секунды для перестроение кеша | 5-15 минут | Можно вернуть no-bid, но нельзя долго терять RTB-контур |
| Campaign | 5 минут | 30 минут | Изменения кампаний важны, но RTB может временно работать на снапшотах |
| Statistics | До 1 минуты при Kafka replication | 30-60 минут | События можно дообработать из Kafka |
| Analytics | 1-24 часа | 4 часа | Не влияет на bid response напрямую |
| Finance | 0-1 минута | 15-30 минут | Финансовые данные критичны и требуют минимальной потери |

## PostgreSQL

- master-slave в разных ДЦ;
- ежедневные полные бэкапы и более частые инкрементальные;
- регулярные учения по восстановлению данных;
- отдельные политики бэкапа для Finance и Campaign.

## Redis

- Redis Cluster с репликацией;
- после полного сбоя Redis выполняется прогрев кеша;
- Bidding Service в период восстановления возвращает no-bid для неизвестных
  кампаний.

## ClickHouse

- восстановление аналитики не должно блокировать RTB и финансы.

## Disaster recovery

Порядок восстановления:

1. Finance PostgreSQL.
2. Campaign PostgreSQL.
3. Kafka cluster.
4. Redis cache warming.
5. Bidding Service.
6. Statistics и Analytics.
7. Dashboard reporting.

Для промежуточного решения допускается один высокодоступный кластер внутри зоны.

## 12-factor соответствие

- Конфигурация через настройку окружения, без прописывания конфигурации в коде.
- Stateless application для Bidding, Gateway, Statistics.
- Централизованный сбор логов.
- Build/release/run разделены через CI/CD.
- Backing services подключаются через конфигурацию.
- Graceful shutdown для потребителей и шлюза.
- Горизонтальное масштабирование при помощи добавления реплик.
- Health checks и readiness probes для всех сервисов.
