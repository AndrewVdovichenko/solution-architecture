# Требования, ограничения и пробелы RetailCore

## Подтвержденные функциональные требования

| Область | Требование |
|---|---|
| Оформление заказа | Система должна поддерживать оформление заказа через интернет-магазин, мобильные каналы и маркетплейсы. |
| Расчет стоимости | При оформлении должны учитываться товары, скидки, промокоды, региональные правила, канал продаж, логистика. |
| Оплата | Система должна создавать платежную сессию и отражать состояние оплаты в заказе. |
| Доставка | Система должна выбирать и резервировать доставку с учетом региона, типа доставки, склада, канала, категории товара и партнера. |
| Статусы заказа | Система должна вести жизненный цикл заказа и поддерживать отклонения: повторная оплата, частичная отмена, ручной разбор, зависшие интеграции. |
| Поддержка | Внутренний пользователь должен видеть, где реально находится заказ, что будет дальше и почему возникло отклонение. |
| Аналитика | Бизнесу нужны отчеты по заказам, статусам, скидкам, логистике и кампаниям с согласованной трактовкой данных. |
| Интеграции | Платформа должна обмениваться данными с платежными провайдерами, службами доставки, складами, маркетплейсами. |

## Архитектурно значимые нефункциональные требования

| Требование | Подтвержденная формулировка | Архитектурное влияние |
|---|---|---|
| Непрерывность бизнеса | Нельзя останавливать бизнес и делать одномоментную миграцию. | Возможна только поэтапная трансформация. Необходимы механизмы отката и фича флаги. |
| Устойчивость ключевых операций | Процессы оплаты, оформления заказа и обещание доставки — это зоны высокого риска. | Не трогаем ядро (core order create, payment orchestration) на первом этапе. |
| Предсказуемость изменений | Бизнесу нужно заранее понимать, сколько это стоит и каковы побочные эффекты при внесении любого изменения. | Нужны границы ответственности, контракты, трассируемость зависимостей. |
| Наблюдаемость | Сейчас цепочку заказа трудно восстановить между монолитом, .NET, Python и Oracle. | Внедрение единого Correlation ID для всех транзакций. Это позволит собрать полный журнал событий, а также создать дашборды и системы алерта на уровне заказа. |
| Прозрачность для поддержки | Карточка заказа не является единым источником правды. | Необходимо разработать модель, которая будет отображать полную историю заказа, включая причины всех отклонений и изменений статуса. |
| Согласованность данных для бизнеса | Разные витрины дают разные цифры по одному процессу. | Нужен канонический слой событий и правила трактовки статусов, скидок, отмен и возвратов. |
| Производительность | Основная БД Oracle становится узким местом из-за длинных транзакций, аналитика нагружает операционную БД. | Поднимаем механизм разгрузки операций чтения. Постепенно снижаем зависимость от прямых запросов к операционной БД. |
| Импортонезависимость | Нужно готовиться к постепенному отказу от Oracle и других зарубежных компонентов. | Нельзя строить новые зависимости вокруг Oracle. Нужен план декомпозиции данных. |

## Ограничения

| Ограничение | Тип | Влияние |
|---|---|---|
| Нет окна для большой остановки платформы. | Бизнес / эксплуатация | Исключает резкий переход (big bang migration). |
| Платформа развернута в собственном ЦОД на физических серверах, ресурсы близки к пределу. | Инфраструктура | Ограничивает быстрый горизонтальный рост и требует аккуратного пилотирования нагрузки. |
| Oracle является операционной БД, содержит бизнес-логику, справочники и интеграционные таблицы. | Технология / данные | Нельзя быстро заменить БД или вынести сервис без инвентаризации процедур и потребителей. |
| Логика статусов, промо и доставки распределена между кодом, Oracle, конфигами и ручными действиями. | Архитектура | Требуется сначала восстановить фактическое поведение. |
| Каналы продаж проходят разные ветки поведения. | Домен | Нельзя проектировать систему только для оптимального пути пользователя. Нужно проработать все возможные вариации поведения, характерные для различных каналов продаж. |
| Внешние партнеры имеют разные способы интеграции и разную надежность подтверждений. | Интеграции | Требуется обязательная реализация механизмов идемпотентности, дедупликации, повторных попыток и строгий контроль семантики передаваемых сообщений. |

## Допущения

| Допущение | Как проверять |
|---|---|
| Первый этап трансформации должен снижать риск и повышать прозрачность, без резких изменений процесса оформления заказа. | Подтвердить у бизнес-спонсора и техлида монолита. |
| Наиболее безопасные первые изменения: внедрение модели событий заказа, создание читаемой модели данных для поддержки и аналитики, повышение наблюдаемости и установку правил доступа к внешним интеграциям. | Проверить с командами заказов, логистики, аналитики и эксплуатации. |
| Для части сценариев допустима eventual consistency, если состояние явно видно поддержке и клиенту. | Уточнить у бизнеса, поддержки, платежей и логистики по сценариям. |

## Архитектурно значимые пробелы

| Пробел | Почему важно | Кто должен уточнить |
|---|---|---|
| RTO/RPO для оформления заказа, оплаты, доставки, статусов, аналитики. | Нужны для проектирования отказоустойчивости и механизмов отката. | Бизнес-спонсор, эксплуатация, владельцы заказов и платежей. |
| SLA/SLO по оформлению заказа, созданию платежной сессии, подтверждению доставки, обновлению статуса. | Нужны для архитектурных порогов, алертов и cutover-критериев. | Бизнес-спонсор, команда заказов, логистика, платежи. |
| p95/p99 задержка в оформлении заказа и допустимая задержка внешних вызовов. | Нужны для решения, какие вызовы оставлять синхронными. | Техлид монолита, эксплуатация. |
| RPS/TPS, пики распродаж, сезонные профили нагрузки. | Нужны для планирования мощности ЦОД и оценки влияния пиковых нагрузок на новые модели. | Эксплуатация, бизнес, команда заказов. |
| Правила деградации промо, доставки и оплаты. | Сейчас есть план для фолбэка, но мы должны убедиться, что этот план полностью соответствует бизнес-требованиям. | Бизнес-спонсор, заказы, платежи, логистика. |
| Требования к консистентности статусов, оплаты, резерва и доставки. | Определяют транзакционные границы и события. | Заказы, платежи, логистика, поддержка. |
| Владельцы канонической модели заказа и статуса. | Без владельца новая модель станет еще одной витриной. | Архитектурный комитет / владельцы заказов, аналитики и поддержки. |
