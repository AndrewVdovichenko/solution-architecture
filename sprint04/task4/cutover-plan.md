# Перевод поддержки и пилотной аналитики на Order Status Read Model

## 1. Цель переключения

Перевести ограниченную группу операторов поддержки и один пилотный аналитический сценарий с ручного сбора статуса заказа из Oracle, логов, админок и смежных команд на `Order Status Read Model`.

Переключение не меняет `createOrder`, платёжную оркестрацию, маршрутизацию доставки и старые источники данных. Старый способ просмотра остается доступен для возможного отката и сверки данных.

## 2. Условия запуска

| Условие | Как проверяется | Владелец |
|---|---|---|
| Новая модель данных работает в тестовом режиме и корректно собирает историю заказов. | Сверочные отчёты по пилотному набору заказов. | Команда заказов, аналитика |
| Для пилотных заказов есть correlation id. | Проверка логов и дашбордов. | Эксплуатация |
| Согласованы правила отображения статусов и причин отклонений. | Согласована и утверждена таблица соответствий: технический статус -> внутренний статус -> клиентское объяснение. | Заказы, поддержка |
| Операторы пилотной группы обучены новому просмотру истории заказа. | Чек-лист обучения и тестовые кейсы. | Поддержка |
| Старые источники доступны для отката на случай сбоя. | Проверка доступа к Oracle/админке/старым отчетам. | Эксплуатация |
| Определены условия для отмены пилота. | Список качественных условий остановки согласован владельцами. | Бизнес, поддержка, эксплуатация |

## 3. Роли и ответственность

| Роль | Ответственность |
|---|---|
| Руководитель миграции | Координирует переключение, принимает решение переходим или не переходим. |
| Команда заказов | Подтверждает корректность событий и статусов заказа. |
| Аналитика | Проверяет сверку Read Model с Oracle и пилотный отчёт. |
| Поддержка | Проверяет применимость истории заказа для реальных обращений. |
| Эксплуатация | Следит за дашбордами, логами, нагрузкой Oracle и потоком событий. |
| Логистика / платежи | Дежурят для разбора статусов доставки и оплаты в пилотном объеме. |
| Бизнес-спонсор | Подтверждает приемлемость операционного риска. |

## 4. Пошаговый сценарий

| Шаг | Действие | Критерий успеха|
| :--- | :--- | :--- |
| **1. Заморозка изменений** | Временно останавливаем любое внесение изменений в правила классификации статусов. | Убеждаемся, что в период перехода нет незапланированных изменений в этой части системы. |
| **2. Фиксация начального состояния** | Мы создаем слепок всех данных по пилотным заказам в старой системе (Oracle) и в новой модели. | Созданный снимок сохранён и готов для сравнения в любой момент времени. |
| **3. Проверка healthcheck** | Проверяем ключевые показатели: насколько быстро новая система обрабатывает события, не падает ли система обработки, стабилен ли объём нагрузки на Oracle. | Все технические показатели находятся в безопасном, согласованном заранее диапазоне. |
| **4. Внедрение для тестовой группы** | Показываем новую систему только узкой группе операторов поддержки. | Эта пилотная группа видит историю заказа, которую генерирует новая модель. |
| **5. Режим сравнения** | Старые источники данных (Oracle/админка) остаются доступными в режиме "только для чтения", чтобы поддерживать параллельный контроль. | Оператор может самостоятельно сравнить информацию из старой и новой систем. |
| **6. Тестирование в реальных условиях** | Мы даем пилотной группе решать реальные проблемы и работать с живыми заказами. | Фиксируем все случаи, когда новая система не смогла предоставить информацию или предоставленные данные были противоречивые. |
| **7. Проверка отчётности** | Мы запускаем один ключевой отчет на новой модели и сравниваем его результат со старым отчетом. | Новая отчётность даёт результат, который совпадает со старым. |
| **8. Итоговая сверка** | В конце рабочей смены проводится глубокий аудит: ищем расхождения и определяем, почему они возникли. | Все расхождения классифицированы: это была ошибка в правилах, задержка, технический сбой или противоречие в старых данных. |
| **9. Решение о дальнейшем пути** | Основываясь на результатах, принимаем решение: закрепляем новую систему, продлеваем период тестирования или полностью отключаем и откатываемся к старой. | Решение принимается всеми ответственными сторонами. |

## 5. Проверки после переключения

| Проверка | Критерии успеха | Критерии провала |
|---|---|---|
| Полнота истории заказа | Оператор видит ключевые изменения статуса, оплаты, доставки и причины отклонений. | Для ответа клиенту нужно снова идти в логи или к смежной команде. |
| Сверка с Oracle | Расхождения объяснены известными правилами задержки или правилами классификации статусов. | Есть неклассифицированные расхождения по критичным состояниям. |
| Наблюдаемость | По order id видны события, обработчики и ошибки. | Нельзя восстановить путь пилотного заказа. |
| Пилотный отчёт | Аналитический отчёт воспроизводимо строится из нового источника. | Показатели расходятся без согласованного объяснения. |
| Нагрузка | Новый контур не создает недопустимой нагрузки на Oracle или поток событий. | Наблюдается деградация ядра или рост ошибок потребителя данных. |

## 6. Условия для остановки пилота

Численные пороги отсутствуют во входных материалах, поэтому до промышленной эксплуатации их нужно согласовать отдельно. До этого используются качественные условия остановки:

- оператор не может ответить по пилотному заказу без возврата к старым источникам;
- расхождение между read model и Oracle затрагивает статус заказа. Расхождение не классифицировано;
- по order id невозможно восстановить цепочку событий;
- поток событий или потребитель данных задерживает обновление так, что поддержка видит устаревшее состояние в рабочем сценарии;
- новый контур влияет на стабильность процесса заказа, оплаты или доставки;
- пилотный аналитический отчёт дает бизнес-вывод, противоречащий старому источнику без объяснения.

## 7. Откат

| Шаг | Действие | Проверка |
|---:|---|---|
| 1 | Отключить новый источник для пилотной группы поддержки через фича флаг. | Операторы снова используют старый процесс. |
| 2 | Вернуть пилотный аналитический отчет на старый источник. | Отчёты продолжают строиться точно так, как были до начала миграции. |
| 3 | Оставить новую систему (поток событий и модель истории) работать, если она не касается критически важных операций | Нет влияния на оформление заказа, оплату, доставку. |
| 4 | Зафиксировать каждый сбой, который произошёл во время перехода. Классифицировать его причину. | Есть список проблем: ошибка в правилах, задержка данных, технический дефект, проблема в интерфейсе или проблема в старых данных. |
| 5 | Обновить критерии готовности перед повторной миграцией. | Согласовано, что нужно исправить, прежде чем мы запустить миграцию снова. |

## 8. Коммуникации

| Момент | Кому | Что сообщить |
|---|---|---|
| До старта миграции | Поддержка, аналитика, заказы, эксплуатация, бизнес-спонсор | Границы пилота, роли, время, поведение во время сбоя. |
| Во время миграции | Участникам перехода | Статус шагов, расхождения, решения продолжаем или нет. |
| После миграции | Все участники | Итоги, проблемы, решение закрепить/продлить/откатить пилот. |
